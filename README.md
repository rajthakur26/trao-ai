# 🌍 Trao.ai — AI Travel Planner

A multi-user web app that turns a few trip details into a complete, **day-by-day
itinerary** with a **budget breakdown** and **hotel suggestions** — generated by
an LLM agent and editable both manually and through a conversational **AI
Concierge**.

> Tell it _where_, _how long_, your _budget level_ and _interests_. Get a full
> plan in seconds, then refine it by chatting: _“make day 3 more outdoorsy”_.

## 🧱 Tech Stack & Justification

| Layer | Choice | Why |
|---|---|---|
| Frontend | **Next.js 14 (App Router) + Tailwind CSS** | Matches the preferred stack; App Router gives clean file-based routing; Tailwind for fast, consistent, responsive UI. |
| Backend | **Node.js + Express** | Matches the preferred stack. Kept as a **separate service** for clean separation of concerns and independent scaling. |
| Database | **MongoDB + Mongoose** | Itineraries are nested, document-shaped data (days → activities) — a natural fit for a document store. Mongoose adds schema validation. |
| LLM | **Groq (`llama-3.3-70b-versatile`)** | Extremely fast inference + a generous free tier, with an OpenAI-compatible JSON mode. Wrapped behind an adapter so the provider is swappable. |
| Language | **JavaScript (ESM)** | Per the assessment; kept clean with a layered architecture. |

## 🏗️ High-Level Architecture

```
 Browser (Next.js / Vercel)
        │  fetch + Bearer JWT
        ▼
 Express API (Render)
        ├── Auth middleware  → verifies JWT, loads user
        ├── Controllers      → validate (Zod) → enforce ownership
        ├── LLM service      → Groq (JSON mode) → Zod-validate → fallback to mock
        └── Mongoose         → MongoDB Atlas (per-user documents)
```

**Request lifecycle for protected routes:** `requireAuth` → `validateBody(zod)`
→ controller loads the trip with `{ _id, owner }` (ownership enforced) → mutate
→ persist → return the updated trip. Errors bubble to a single
`errorHandler` that returns clean JSON.

## 🔐 Authentication & Authorization

- **Registration / login** issue a **JWT** (`sub = userId`), signed with
  `JWT_SECRET`. Passwords are hashed with **bcrypt** and the hash is
  `select: false` so it never leaves the database by accident.
- The token is sent as an `Authorization: Bearer <token>` header. `requireAuth`
  verifies it and attaches `req.user`.
- **Data isolation** is the core guarantee: every trip read/write loads the
  document with a combined filter `Trip.findOne({ _id, owner: req.user._id })`.
  Another user's trip is therefore **indistinguishable from a non-existent one
  (404)** — no existence leak, and the check lives in one reusable helper
  (`getOwnedTrip`) that every mutating handler uses.
- This is verified by an end-to-end test: User B requesting User A's trip gets
  `404`, and B's trip list is empty.

**Trade-off:** I store the JWT in `localStorage` and send it as a Bearer token
rather than using httpOnly cookies. Reason: the frontend and backend live on
**different domains** (Vercel ↔ Render); Bearer tokens avoid cross-site cookie
(`SameSite=None`) friction and CORS-credentials complexity. The trade-off is
slightly higher XSS exposure, mitigated by React's escaping, `helmet`, strict
CORS, and short-lived tokens.

## 🤖 AI Agent Design

The agent lives in [`server/src/services/llm/`](server/src/services/llm/) and is
built around a **“constrain, validate, or fall back”** philosophy so malformed
model output can never corrupt the database or break the UI.

**Three capabilities:**
1. **`generatePlan`** — full itinerary + budget + hotels from the trip inputs.
2. **`regenerateDay`** — regenerates a single day given a free-text instruction,
   keeping the rest of the trip consistent.
3. **`runConcierge`** — interprets a natural-language message and returns a
   **structured list of operations** (`add_activity` / `remove_activity` /
   `regenerate_day`) that the backend applies safely.

**How it's made robust:**
- **System prompt** fixes persona + hard rules (budget realism per level,
  3-5 activities/day, currency).
- **Groq JSON mode** (`response_format: json_object`) forces parseable output.
- **Zod schemas** validate every response; the **budget total is recomputed**
  server-side; the **day count is normalised** to exactly what the user asked.
- **Mock fallback** — if `GROQ_API_KEY` is missing _or_ a live call fails, a
  deterministic generator produces the same JSON shape, so the app stays fully
  functional (great for demos/offline). The API response includes
  `meta.source: "groq" | "mock"` for transparency.
- **Localised currency** — budgets/prices default to **Indian Rupees (₹, INR)**
  since the target users are in India; the agent is prompted for realistic
  Indian-market costs and the UI formats with lakh grouping. Configurable via
  `DEFAULT_CURRENCY`.

**Why the concierge returns operations instead of mutating directly:** the LLM
never touches the database. It only _proposes_ structured edits; the server
validates and applies them. This keeps authorization, data integrity, and
business logic firmly on the trusted server.

## 💡 Creative Feature — AI Concierge + Shareable Trips

I built **two** complementary custom features:

### 1. AI Concierge (conversational editing)
Instead of hunting through buttons, the user just **chats**: _“remove the museum
visits”_, _“add a fancy dinner on the last day”_, _“make day 2 adventurous”_.
The agent replies in natural language **and** applies the real edits, showing a
checklist of exactly what changed.

**Problem it solves:** editing a structured itinerary with forms is tedious. A
concierge turns multi-step edits into one sentence — this is the “modify
dynamically” requirement, elevated into a genuinely agentic UX. **Engineering
judgment:** the agent returns *operations*, not free-form state, so the server
stays the single source of truth and edits remain authorized and validated.

### 2. Public share links
Any trip can be published as a **read-only public link**
(`/shared/<token>`) — viewable without an account, with the owner field
stripped. **Problem it solves:** people plan trips _with_ others; this enables
sharing without weakening per-user data isolation (it's opt-in, revocable, and
exposes no private fields).

## 🚀 Getting Started (Local)

**Prerequisites:** Node 18+, a MongoDB instance (local or Atlas), and
optionally a free [Groq API key](https://console.groq.com/keys) (works without
one via the mock).

```bash
# 1. Backend
cd server
cp .env.example .env          # then edit values (MONGODB_URI, JWT_SECRET, GROQ_API_KEY)
npm install
npm run dev                   # → http://localhost:5000  (GET /api/health to verify)

# 2. Frontend (new terminal)
cd client
cp .env.local.example .env.local   # NEXT_PUBLIC_API_URL=http://localhost:5000
npm install
npm run dev                   # → http://localhost:3000
```

Open http://localhost:3000, register, and create a trip.

> **No Groq key?** Leave `GROQ_API_KEY` blank — the app runs in deterministic
> mock mode and every feature still works.

## ⚖️ Key Design Decisions & Trade-offs

- **Separate Express backend** over Next API routes → cleaner separation and
  matches the preferred stack, at the cost of two deploy targets.
- **Bearer token in localStorage** over httpOnly cookies → simpler cross-domain
  setup (see Auth section), accepting marginally higher XSS surface.
- **Validate-or-fallback LLM layer** → the app never breaks on a bad model
  response; the price is extra normalisation code.
- **Operations-based concierge** → safety and auditability over letting the
  model rewrite state directly.

## 🚧 Known Limitations

- LLM cost/budget estimates are **AI approximations**, not live pricing.
- No automated test suite is committed (a scripted E2E check was used during
  development); production would add Jest/Supertest + Playwright.
- Render's free tier cold-starts (~30s) on first request after idle.
- Itinerary regeneration is synchronous; very long trips could be moved to a
  background job + websockets for progress.
